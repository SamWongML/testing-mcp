**Handoff notes (P8 — 2026-07-24):**

- **Shape.** The async surface is enabled only when a run database is configured (`ctx.db`);
  without one the server is the P7 sync-only surface unchanged (verified: sync boot exposes
  exactly `list_tests`/`describe_test`/`run_test`/`get_report`/`list_runs`, no `tasks`
  capability). Async execution is inherently durable, so everything rides P6's Postgres
  `jobs` queue + `tasks` hot-state table — no broker (ADR-004).

- **Files (all in `packages/mcp-server/src/`):**
  - `execute.ts` — `executeEntry(ctx, entry, {params,env,signal,runId,onProgress})`: the shared
    test/suite executor (imports the authored def, `expandUnits` for the cell, dispatches
    `runTest`/`runSuite`). Used by both the inline `run_test` and the worker. `tools.ts`'s old
    private `executeInline` was folded into it (DRY).
  - `tasks.ts` — lifecycle glue over `@atp/store`: `submitRun` (atomic **create task row +
    enqueue job** in one tx, with idempotency), `getRun`, `getRunResult` (loads the terminal
    trace), `cancelRun` (flags job+task), `parseSpec`, `resultStateFor` (ExecutionStatus →
    TaskState), `enqueueRun`, `DEFAULT_TASK_TTL_MS`.
  - `worker.ts` — `runClaimedJob` (execute one claimed job to terminal under an AbortSignal;
    heartbeat+cancel-poll timer; serialized progress writes before the terminal write),
    `claimAndRun`, `reapOnce`, `startWorker` (the `MODE=worker` loop) → `WorkerHandle.stop()`.
  - `sdk-tasks.ts` — `SdkTaskStore implements TaskStore` (the SDK's experimental
    `TaskStore`): the bridge between the MCP Tasks protocol and the **same** durable
    `tasks`/`jobs` rows the worker uses. `createTask` mints the row + enqueues atomically
    (deriving the spec from the augmented request's `arguments`); `getTask`/`getTaskResult`
    project the durable row / trace onto the SEP-1686 `Task`/`CallToolResult`;
    `updateTaskStatus('cancelled')` → `cancelRun`. Keyed by `runId == taskId`, so it works
    across the stateless fresh-server-per-request lifecycle (no session affinity).
  - `task-tools.ts` — `run_suite` (task-augmented via
    `server.experimental.tasks.registerToolTask`, `taskSupport:'required'`), `run_selection`
    (plain batch submit), and the `get_run`/`get_run_result`/`cancel_run` mirror tools.
  - `main-worker.ts` — the `MODE=worker` entrypoint (`pnpm dev:worker`); fails fast without
    `DATABASE_URL`.
  - `server.ts` — `buildMcpServer` now enables the async surface + `taskStore` +
    `capabilities.tasks` when `ctx.db` is present.

- **Engine change (only one, minimal, stays pure).** Added `onProgress?({completed,total,
  nodeId})` to `RunOptionsBase`, fired as each node/step settles in `runTest`/`runSuite`. The
  worker maps it to `taskStore.setProgress` for live k/n progress. No MCP/schema coupling.

- **Store change.** Added a `created_at` column to `tasks` (needed for a valid SEP-1686
  `Task.createdAt`; `ttl` is derived as `expiresAt − createdAt`) and `updatedAt` on
  `TaskRecord`; added `PostgresTaskStore.create()` (insert-only, `onConflictDoNothing`) — the
  atomic primitive idempotent submission builds on. `created_at` was added to the existing
  `0000_init.sql` (greenfield; test DBs recreate from scratch each run).

- **Idempotency.** `submitRun({idempotencyKey})` uses the key as the `runId` and `create()`s
  the task row insert-only inside the enqueue transaction: a repeat key finds the row already
  present → returns the existing run (`deduped:true`), enqueues nothing. This collapses the
  §16.2 `idempotency` table into the `tasks` PK for stage-1 (the P11 Dynamo adapter gets its
  own `idempotency` table).

- **SDK Tasks augmentation — verified against the installed 1.29.0 source (experimental, "may
  change without notice"), not memory.** The full flow was de-risked with a spike before
  building: server advertises `tasks:{cancel,requests:{tools:{call}}}` + a `taskStore`;
  `run_suite` is registered via `registerToolTask`; a client drives it with
  `client.experimental.tasks.callToolStream(..., {task:{ttl}})` → `taskCreated` → poll
  `tasks/get` → on `completed` the SDK fetches `tasks/result` (→ our `getTaskResult`) →
  `result`; `cancelTask` → `tasks/cancel` → our `updateTaskStatus('cancelled')`. Key SDK
  facts confirmed in `node_modules/@modelcontextprotocol/sdk`: `ProtocolOptions.taskStore`
  auto-enables the `tasks/*` handlers; the client's `requestStream` convenience wrapper fetches
  `tasks/result` only on `completed` (failed/cancelled yield an error), so the worker never
  needs `storeTaskResult`. **Note:** the *server's* generic `tasks/result` handler invokes
  `getTaskResult` for *any* terminal status (a client can call `getTaskResult()` directly), so
  `getTaskResult`/`getRunResult` handle the trace-less terminal cases explicitly (see below).

- **Trace-less terminal results (fixed in review).** Two terminal paths never call `persistRun`
  and so write no trace: a job cancelled while still queued, and `finalizeError` (bad spec /
  unknown entry / pre-execution throw). `getRunResult`/`SdkTaskStore.getTaskResult` key off
  `task.resultRef` (set only on the `persistRun` paths): when it is null they return a
  `ready` result with no `ExecutionResult`, surfacing `task.error`, instead of throwing a
  misleading "No run" from the trace loader. Covered by two pg-gated tests.

- **Scope decisions (deliberate, coherent, minimal).**
  - SDK task augmentation is wired for **`run_suite`** (the flagship one-run-one-task case).
    `run_selection` is a **plain** batch tool (multiple runs don't fit the one-task-one-result
    model) and `run_test`'s long-running auto-task uses the **plain** durable path — both
    poll via the mirror tools. All three share the one durable store, so a `run_suite` task
    and a mirror-tool `get_run` of the same `runId` observe identical state (§8.2 graceful
    degradation).
  - `run_test` now **auto-tasks** a long-running test (enqueue + return `{runId,state}`)
    instead of rejecting it; without a db it errors pointing at the async/database requirement.
  - Cancellation is SEP-1686 flag-based: `cancel_run`/`tasks/cancel` set the job+task cancel
    flags; the worker aborts between/within nodes (undici `AbortSignal`) and writes the
    terminal `cancelled` state. A queued (unclaimed) job is finalized `cancelled` at claim
    time.

- **Testing posture (read this).** The DB-backed paths (`tasks.ts`, `worker.ts`,
  `sdk-tasks.ts`, the tool wiring) are covered by `async-lifecycle.test.ts` — **pg-gated, so
  they skip offline** and were **not run against a live Postgres this session** (Docker/pg
  unavailable here), same posture P6/P7 documented. Verify under `ATP_TEST_DATABASE_URL`. What
  *was* really exercised locally (real red→green): the engine `onProgress` callback
  (`runner.test.ts`/`suiteRunner.test.ts`) and `execute.ts` running a **suite** end-to-end
  offline incl. progress + signal cancellation (`execute.test.ts`). The SDK Tasks mechanics
  were proven by the spike. Gate: typecheck / lint / test (338 passed | 36 skipped) / compile
  all green.

- **Minor known race (documented, benign for stage-1).** Live progress writes are serialized
  and awaited before the terminal write, so `progressPct` can't be clobbered below 100 by the
  worker's own ticks. `updateTaskStatus('cancelled')` sets flags but does not force terminal
  state; the worker owns the terminal transition (a queued-then-cancelled job is finalized at
  claim time).
